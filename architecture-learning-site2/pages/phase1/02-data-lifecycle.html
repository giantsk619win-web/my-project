🌱1-2. データのライフサイクル

フクロウ先生の導入

このサイトのテーマは「データの旅」。でも、データって一体どこから生まれて、どこへ行くのでしょう？実は、私たちが日常で体験していることと同じなんです。たとえば、あなたがコンビニでお弁当を買う——その瞬間に「購買データ」が生まれ、集計され、分析され、「次にどんな商品を仕入れるか」という経営判断に使われます。データには一生（ライフサイクル）があるんですよ。

データの旅路：6つのステージ

1つのデータが生まれてから役目を終えるまで、大きく6つのステージを旅します。郵便物にたとえながら見ていきましょう。

ステージ やること 郵便のたとえ 代表的な技術
1. 生成 データが生まれる 手紙を書く Webフォーム、IoTInternet of Things。センサーやカメラなど、インターネットに接続されたモノのことセンサー、アプリ入力
2. 収集 データを集める ポストに投函→集配局へ APIApplication Programming Interface。異なるシステム間でデータをやり取りするための仕組み、ETLExtract（抽出）、Transform（変換）、Load（読み込み）の略。データを別のシステムに移す処理ツール、ログ収集
3. 処理 データを加工・変換する 仕分け・消印押し バッチ処理、ストリーム処理データが発生するたびにリアルタイムで処理する方式。バッチ処理（まとめて処理）の対義語、データクレンジング
4. 保存 データを安全に蓄積する 届け先の郵便受けに届く RDBRelational Database。表形式でデータを管理するデータベース。Oracle、PostgreSQL、MySQLなどが代表的、NoSQLNot Only SQL。表形式以外の方法でデータを格納するデータベース。大量データや柔軟な構造に強い、クラウドストレージ
5. 活用 データから価値を引き出す 手紙を読んで行動する BIBusiness Intelligence。蓄積されたデータをグラフや表にして経営判断に活用する仕組みツール、ダッシュボード、AI/機械学習
6. 廃棄 不要になったデータを安全に消す 古い手紙をシュレッダー データ保持ポリシー、アーカイブすぐには使わないが保存義務があるデータを、低コストのストレージに移して保管すること、安全消去

ポイント

お客様の案件を考えるとき、「このデータはどのステージの話をしているんだろう？」と意識するクセをつけましょう。「データを集めたい」のか「集めたデータを分析したい」のかで、必要な技術も費用もまったく違ってきます。

業界別：データの旅の実例

では、実際の業界ではどんなデータがどんな旅をしているのでしょうか？お客様との会話で役立つ3つの例を見てみましょう。

小売業：購買データの旅

生成：お客様がレジでバーコードをスキャン → 収集：POSデータが本部サーバーに集約 → 処理：日次バッチで売上集計 → 保存：データウェアハウスに蓄積 → 活用：BIツールで売れ筋分析、需要予測で仕入れ最適化 → 廃棄：法定保存期間経過後にアーカイブ

製造業：IoTセンサーデータの旅

生成：工場の機械に取り付けたセンサーが温度・振動を計測 → 収集：IoTゲートウェイ経由でクラウドに送信 → 処理：リアルタイムで異常値を検知 → 保存：時系列データベースに蓄積 → 活用：故障予兆の検知、保守計画の最適化 → 廃棄：一定期間後に詳細データを削除し、集計結果のみ保持

金融業：取引データの旅

生成：顧客がネットバンキングで振込操作 → 収集：リアルタイムで基幹システムへ → 処理：即時の残高更新と不正検知チェック → 保存：高信頼性のRDBに厳重保管 → 活用：マネーロンダリング対策の分析、顧客の資産運用提案 → 廃棄：法規制に基づき10年以上保存した後にアーカイブ

営業が知っておくべき各ステージの技術キーワード

すべてを深く理解する必要はありません。「この言葉が出てきたら、データの旅のどのステージの話だな」とピンとくるレベルで十分です。

ステージ よく出るキーワード お客様の言い方（例）
生成 フォーム入力、スキャン、センサー、ログ 「店舗のデータを取りたい」
収集 API連携、データ連携、インポート、ETL 「各拠点のデータを本部に集めたい」
処理 バッチ、リアルタイム、変換、クレンジング 「データをきれいにして使えるようにしたい」
保存 DB、ストレージ、データレイク、バックアップ 「安全にデータを保管したい」
活用 BI、ダッシュボード、AI、レポート 「データを見える化したい」
廃棄 保持ポリシー、アーカイブ、GDPR対応 「古いデータの扱いに困っている」

営業の現場では...

お客様がDX推進を検討中で「うちもデータ活用をやりたい」と言ったとき、「何のデータを、どう集めて、どう使いたいですか？」とライフサイクルに沿って質問すると、漠然とした要望を具体的な要件に落とし込めます。「データを見える化したい」は活用ステージの話ですが、実はその前に収集と処理の仕組みが必要——こうした全体像を押さえておくと、適切な規模感と予算感で提案できるようになります。

開発チームの視点

開発チームは、データのライフサイクル全体を考えてアーキテクチャを設計します。とくに重要なのが「データ量の見積もり」と「処理のリアルタイム性」。営業さんからよくある質問で助かるのが「1日にどのくらいのデータが発生しますか？」「リアルタイムで見たいですか、翌日でもOKですか？」——この2つの情報だけで、アーキテクチャの方向性がかなり絞り込めます。逆に、この情報がないと見積もりが大幅にブレてしまうんです。

理解度チェック

Q1

データのライフサイクル

お客様が「各店舗の売上データを本部で一括管理したい」と言っています。これはデータのライフサイクルのどのステージに関する要望ですか？

  • A 生成
  • B 収集
  • C 活用
  • D 廃棄

「各拠点のデータを本部に集める」は、データの収集ステージの話です。分散したデータを集約する仕組み（API連携やETLツール）が必要になります。データの生成（各店舗でのPOSデータ記録）はすでに行われている前提ですね。

Q2

ETLとは

ETL（Extract, Transform, Load）は、データの旅のどの段階で主に使われる技術ですか？

  • A 生成と廃棄
  • B 活用のみ
  • C 収集と処理
  • D 保存のみ

ETLは「抽出（Extract）→ 変換（Transform）→ 読み込み（Load）」の略で、データを別のシステムから取り出し、必要な形に変換して、保存先に格納する処理です。つまり、収集と処理にまたがる技術です。データ活用基盤の提案では頻出するキーワードですよ。

Q3

見積もりに効く質問

データ活用の案件で、開発チームが「見積もりに必要」としてとくに重視する情報はどれですか？

  • A 1日のデータ量と、リアルタイム性の要否
  • B お客様の社長の名前と年齢
  • C 競合他社が使っているツール名
  • D お客様のオフィスの所在地

開発チームがアーキテクチャを検討する際、データ量とリアルタイム性は最も重要な判断材料です。「1日に数百件のデータで翌日見られればOK」と「1秒間に数万件のデータをリアルタイムで処理したい」では、必要な技術もコストもまったく異なります。この2つを聞いてくれる営業さんは、開発チームから「わかってるな！」と信頼されますよ。

データの旅路、全体像がつかめましたか？大切なのは、お客様の話を聞いたとき「それはデータの旅のどのステージの話だろう？」とイメージできること。次のトピックでは、その旅を支える「アーキテクチャパターン」を学びます。モノリスやマイクロサービスなど、名前だけは聞いたことがあるかも？身近なたとえで整理していきましょう！
