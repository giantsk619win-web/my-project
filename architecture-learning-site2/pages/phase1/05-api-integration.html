🌱1-5. API・システム連携の基礎

フクロウ先生の導入

企業のシステムは1つだけで完結することはほとんどありません。販売管理システム、在庫管理システム、会計システム...これらがバラバラに動いていたら大変ですよね。「基幹システムとWebサイトをつなぎたい」「SaaSと自社システムを連携させたい」——こんな要望は日常茶飯事。今日学ぶのは、異なるシステム同士をつなぐ「通訳者」の役割を果たすAPIApplication Programming Interface。異なるシステム同士がデータをやり取りするための「窓口」の仕組みとシステム連携の基礎です。

APIとは？——「システムの通訳者」

たとえば、あなたが海外旅行に行ったとします。言葉が通じない国で、レストランで注文するには通訳が必要ですよね。APIはまさにこの「通訳者」の役割。異なる言葉（データ形式）を話すシステム同士の間に立って、お互いのやり取りを仲介します。

もっと身近な例で言えば、コンセントもAPIの一種と考えられます。

  • どのメーカーの家電でも、コンセントの形状（＝APIの仕様）が統一されているから差せる
  • 電気の流れ方（＝データ形式）が決まっているから、正しく動く
  • 掃除機もテレビもスマホ充電器も、同じコンセントで使える（＝異なるシステムが連携できる）

3つの主な連携方式

システムをつなぐ方法は大きく3つ。それぞれ、通信の仕方が異なります。

連携方式 たとえ 仕組み タイミング 代表技術
REST APIWebの仕組み（HTTP）を使ったAPI。URLにリクエストを送ると、JSON形式でデータが返ってくる。現在最も普及している方式 電話（すぐ返事がもらえる） システムAがシステムBに「データちょうだい」とリクエストし、すぐに返事が返る リアルタイム HTTP/HTTPS、JSON、RESTful API
メッセージキューシステム間でメッセージを一時的に保管して順番に届ける仕組み。郵便受けのように、受け手の都合に合わせて取り出せる 郵便（後で届く、確実に届く） システムAが「メッセージ」を箱に入れ、システムBが都合のいいときに取り出す 非同期（少し遅れる） Amazon SQS、RabbitMQ、Apache Kafka
ファイル連携CSVやExcelなどのファイルをシステム間で受け渡しする最も古典的な連携方式。単純だが手動作業が入りやすい 回覧板（まとめて渡す） システムAがCSV等のファイルを出力し、システムBがそのファイルを取り込む バッチ（定期的にまとめて） CSV/Excel、SFTP、共有フォルダ

ポイント

どの方式を選ぶかは「リアルタイム性」と「信頼性」と「コスト」のバランスで決まります。「今すぐデータがほしい」ならAPI連携、「確実に届いてほしいが多少遅れてもいい」ならメッセージキュー、「1日1回の一括更新でいい」ならファイル連携が基本的な選択肢です。

連携パターンの使い分け

お客様から具体的な要望を聞いたとき、どの連携方式が適しているかの判断基準を整理しましょう。

お客様の要望 適した方式 理由
「ECサイトの注文が入ったら、即座に在庫システムに反映したい」 REST API リアルタイムでの在庫反映が必要。注文 → 在庫減の即時連携
「注文が増えてもシステムが落ちないようにしたい」 メッセージキュー 大量の注文を一時的にキューに溜めて、処理側が自分のペースで処理できる
「毎晩、販売データを会計システムに取り込みたい」 ファイル連携 日次のバッチ処理で十分。シンプルで既存システムとの相性が良い
「SalesforceのデータをBIツールで分析したい」 REST API + バッチ SalesforceのAPIでデータを取得し、定期的にBIツールに連携

豆知識

最近よく聞く「iPaaSintegration Platform as a Service。クラウド上でシステム連携を簡単に構築できるサービス。プログラミング不要（ノーコード）で連携を設定できるものも多い」は、クラウド上でノーコード/ローコードでシステム連携を作れるサービス。MuleSoftやMicrosoft Power Automate、Zapierなどが有名です。小〜中規模の連携案件では有力な選択肢ですよ。

営業の現場では...

お客様から「システム連携したい」と言われたとき、まず確認すべきは「何と何を、どのくらいの速さで、どの方向に」つなぎたいのかです。「受発注システムとECサイトを、注文発生時にリアルタイムで連携したい」と具体化できれば、SEは「ならREST APIで、こういう構成で...」とすぐに検討に入れます。逆に「なんか連携したい」だけだと、SEも困ってしまいます。連携の要件を具体化する力は、提案のスピードに直結しますよ。

開発チームの視点

システム連携の設計で開発チームが最も気にするのは「エラー時の挙動」です。連携先のシステムが落ちていたらどうする？データが途中で消えたら？こうした「異常系」の設計がコスト全体の半分を占めることもあります。営業さんにお願いしたいのは、「連携が失敗したとき、業務としてどう対応するか」もお客様に確認してくること。「手動で再投入する」のか「自動リトライでいい」のかで、設計の複雑さもコストも大きく変わります。

理解度チェック

Q1

APIとは

APIの役割を最も適切に表しているものはどれですか？

  • A 異なるシステム間でデータをやり取りするための「窓口」
  • B データを暗号化して安全に保管する仕組み
  • C サーバーのCPU使用率を監視するツール
  • D プログラミング言語の一種

APIは「Application Programming Interface」の略で、異なるシステム同士がデータをやり取りするための「窓口」です。通訳者やコンセントにたとえられるように、統一された仕組みを通じて多様なシステムが連携できるようにします。

Q2

連携方式の選び方

「大量の注文が一度に来てもシステムを落としたくない」という要件に最も適した連携方式はどれですか？

  • A REST API（直接呼び出し）
  • B メッセージキュー
  • C ファイル連携（CSV出力→取り込み）
  • D 手動でデータを入力する

メッセージキューは「郵便受け」のように、大量のメッセージを一時的に溜めて、処理側が自分のペースで順番に取り出すことができます。急激なアクセス増でもキューがバッファとなり、後続のシステムが過負荷で落ちることを防げます。REST APIの直接呼び出しでは、大量リクエストが一気に押し寄せると処理しきれない可能性があります。

Q3

連携の要件確認

お客様に「システム連携したい」と言われたとき、最初に確認すべきことはどれですか？

  • A 使うプログラミング言語
  • B サーバーのスペック
  • C APIの技術仕様書
  • D 何と何を、どのくらいの速さで、どの方向につなぎたいか

技術的な詳細（言語、スペック、仕様書）はSEが検討するものです。営業がまず確認すべきは、ビジネス要件——「何と何を」「どのくらいの速さで」「どの方向に」連携したいのか。これが明確になれば、SEはすぐに適切な方式を検討できます。プログラミング言語やサーバースペックは、方式が決まった後の話です。

フクロウ先生の振り返り

API連携の3つの方式——電話（REST API）、郵便（メッセージキュー）、回覧板（ファイル連携）。それぞれの特徴と使い分けがわかりましたね。システム連携は提案の「つなぎ目」になる部分で、ここの要件をしっかり押さえると、開発チームからの信頼度がグッと上がります。
