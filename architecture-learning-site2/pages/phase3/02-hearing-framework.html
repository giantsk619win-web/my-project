🗺️3-2. 課題ヒアリングフレームワーク

フクロウ先生の導入

「技術のことはわからないから、技術的な質問ができない」——よく聞く悩みです。でもね、実は技術的な質問をしなくても、技術要件は引き出せるんです。コツは「お客様のビジネスの言葉」で質問すること。今日は、そんなヒアリングの秘訣を20の質問テンプレートで身につけましょう！

ヒアリング質問テンプレート20問

4つのカテゴリに分けた20の質問です。各質問の下に、お客様の回答から読み取れる技術要件を示しています。

カテゴリA：現状把握（今を知る）

1現在のシステムは何年前に導入されたものですか？

→ 技術要件：レガシー度合い。5年以上ならEOL対応リスク、10年以上ならモダナイゼーション案件の可能性。

2そのシステムを日常的に使っているのは何人くらいですか？

→ 技術要件：同時接続ユーザー数。スケーラビリティ設計の基礎データ。

3一番忙しい時期や時間帯はいつですか？通常と比べてどのくらい増えますか？

→ 技術要件：ピーク倍率。オートスケーリングの必要性判断。

4システムが止まったら、業務にどの程度影響がありますか？

→ 技術要件：可用性要件（SLAレベル）。ビジネスインパクトからRTO/RPOを逆算。

5今のシステムで一番困っていることは何ですか？

→ 技術要件：最優先の課題特定。パフォーマンス問題か、機能不足か、運用負荷か。

カテゴリB：課題特定（困りごとを深掘り）

6そのデータは「今すぐ」見たいですか？それとも「翌日」でも大丈夫ですか？

→ 技術要件：リアルタイム処理 vs バッチ処理の判断。アーキパターン選定の分岐点。

7他のシステムとデータを連携する必要はありますか？どのシステムとですか？

→ 技術要件：API連携/データ連携の要否と範囲。連携先のシステム制約も把握。

8社外の人（取引先、顧客）もこのシステムを使いますか？

→ 技術要件：インターネット公開の要否。セキュリティレベル、認証方式の方向性。

9スマートフォンやタブレットからも使いたいですか？

→ 技術要件：レスポンシブWeb or ネイティブアプリの判断。モバイルBFFの要否。

10今の作業の中で「手作業」でやっていて負担が大きい部分はどこですか？

→ 技術要件：自動化・RPA・ワークフローシステムの要否と優先度。

カテゴリC：要件整理（何が必要かを明確にする）

113年後、5年後にこのシステムの利用者やデータ量はどうなりそうですか？

→ 技術要件：スケーラビリティ設計方針。将来を見据えたアーキ選定。

12新しい機能を追加する頻度はどのくらいになりそうですか？

→ 技術要件：変更容易性。モノリスかマイクロサービスかの判断材料。CI/CDの要否。

13このシステムで扱うデータの中に、個人情報や機密情報はありますか？

→ 技術要件：セキュリティレベル。暗号化要件、アクセス制御、監査ログの要否。

14画面が表示されるまでに何秒くらいなら許容できますか？

→ 技術要件：パフォーマンス要件。レスポンスタイム目標値の設定。

15システムの利用は日本国内だけですか？海外からのアクセスもありますか？

→ 技術要件：マルチリージョン/CDNの要否。データ保管場所の法的制約（GDPR等）。

カテゴリD：制約確認（できないことを把握する）

16ご予算はどのくらいの規模感をお考えですか？

→ 技術要件：アーキの複雑さの上限。クラウド vs オンプレの判断にも影響。

17いつまでに稼働開始したいですか？

→ 技術要件：開発手法の選択（ウォーターフォール vs アジャイル）。段階リリースの要否。

18社内にシステムを運用・管理できる担当者はいらっしゃいますか？

→ 技術要件：運用体制。マネージドサービスの採用度合い。SaaS/PaaSの活用レベル。

19「これだけは使い続けたい」という既存のシステムやツールはありますか？

→ 技術要件：既存システム連携の制約。移行戦略の方向性。API連携の要否。

20セキュリティやコンプライアンスで守るべき基準やルールはありますか？

→ 技術要件：準拠すべき規制（FISC、ISMAP等）。セキュリティ設計の前提条件。

営業の現場では...

20問すべてを聞く必要はありません。商談の流れに合わせて重要な5〜8問を選んで聞くのが現実的です。ポイントは「カテゴリAの現状把握」から入って、自然にBの課題深掘りへ。いきなり予算や納期（カテゴリD）から聞くと身構えられるので、最後に確認する流れがスムーズです。

開発チームの視点

営業さんがこのフレームワークでヒアリングしてきた情報は、開発チームにとって宝の山です。とくに「同時ユーザー数」「ピーク倍率」「リアルタイム性」「セキュリティ要件」の4つが揃っていれば、初期のアーキテクチャ案をすぐに作成できます。完璧でなくていい。「おおよそ100人くらい」でも十分。数字があるだけで設計の精度が格段に上がるんです。

理解度チェック

Q1

ヒアリングの優先順位

初回ヒアリングで最初に聞くべきカテゴリはどれですか？

  • A予算（カテゴリD：制約確認）
  • B技術スタック（カテゴリC：要件整理）
  • C現行システムの状況（カテゴリA：現状把握）
  • Dセキュリティ基準（カテゴリD：制約確認）

まず現状把握から入るのが基本。今のシステムや業務の状況を理解してから、課題を深掘りし、要件を整理し、最後に制約を確認する流れがスムーズです。いきなり予算やセキュリティの話をすると、お客様が身構えてしまいます。

Q2

技術要件の引き出し方

「そのデータは今すぐ見たいですか、翌日でもいいですか？」この質問で判断できる技術要件は？

  • Aセキュリティレベル
  • Bリアルタイム処理 vs バッチ処理
  • Cクラウド vs オンプレミス
  • Dプログラミング言語の選択

「今すぐ」ならリアルタイム処理、「翌日でOK」ならバッチ処理。この判断はアーキテクチャパターンの選定に大きく影響します。技術的な用語を使わずに、ビジネスの言葉で技術要件を引き出すテクニックです。

Q3

開発チームが最も欲しい情報

開発チームがアーキテクチャ検討のために、営業からとくに欲しい4つの情報の組み合わせとして正しいものは？

  • A同時ユーザー数、ピーク倍率、リアルタイム性、セキュリティ要件
  • Bお客様の売上高、社員数、設立年、資本金
  • C希望するUI色、フォント、ロゴ、アイコン
  • D競合他社のシステム名、費用、納期、ベンダー名

開発チームがアーキテクチャを検討するのに最も必要なのは「同時ユーザー数」「ピーク倍率」「リアルタイム性」「セキュリティ要件」の4つ。これが揃えば初期アーキテクチャ案の作成に着手できます。
