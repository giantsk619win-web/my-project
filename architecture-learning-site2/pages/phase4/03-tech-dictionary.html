4-3. 技術用語ブリッジ辞典

フクロウ先生より

開発チームとの打ち合わせで「？？？」となったこと、ありませんか？この辞典は技術用語を営業の言葉に翻訳する橋渡し（ブリッジ）です。用語の正確な定義ではなく「営業として最低限知っておくべきこと」に絞っています。検索やカテゴリで絞り込んで、必要な用語をサッと確認してください！

サーバー Server
インフラ

データやサービスを提供する側のコンピューター。レストランでいう「厨房」。お客様（クライアント）からの注文を受けて料理（データ）を返す。

開発チーム：「このサーバーのスペックでは足りない」→ コンピューターの処理能力が不足している
Phase 1-1 参照
ロードバランサー Load Balancer
インフラ

アクセスを複数のサーバーに振り分ける装置。デパートの「案内係」のようなもので、空いている窓口に誘導する。

開発チーム：「LBの後ろにWebサーバーを3台置く」→ 負荷分散してアクセス集中に耐える構成
Phase 2-5 参照
CDN Content Delivery Network
インフラ

画像やJSファイルなど、変わらないコンテンツを世界中のサーバーに配置し、ユーザーに近い場所から配信する仕組み。コンビニの配送センターのようなイメージ。

開発チーム：「静的ファイルはCDNに乗せましょう」→ 表示速度が速くなり、サーバーの負荷も下がる
Phase 2-5 参照
DNS Domain Name System
インフラ

ドメイン名（例：example.com）をIPアドレス（サーバーの住所）に変換する仕組み。インターネットの「電話帳」。

開発チーム：「DNSの切り替えに最大72時間かかる」→ ドメインの向き先変更は即座に反映されない
Phase 1-4 参照
ファイアウォール Firewall
インフラ

不正なアクセスを遮断する「防火壁」。許可された通信だけを通し、それ以外はブロックする。

開発チーム：「FWでポートを開ける必要がある」→ 特定の通信を許可する設定変更が必要
Phase 2-4 参照
VPN Virtual Private Network
インフラ

インターネット上に暗号化された専用トンネルを作る技術。外出先から社内ネットワークに安全にアクセスできる。

開発チーム：「VPN経由でないとアクセスできない」→ セキュリティのため専用接続が必要
Phase 2-4 参照
オンプレミス On-Premises
インフラ

自社のサーバールームやデータセンターにサーバーを設置して運用する形態。「自社ビルに自前の厨房を持つ」イメージ。

開発チーム：「現行システムはオンプレです」→ クラウドではなく自社設備で動いている
Phase 1-4 参照
コンテナ Container / Docker
インフラ

アプリケーションと動作に必要なものをひとまとめにした「箱」。どの環境でも同じように動く。引っ越しのダンボール箱のように、中身ごと移動可能。

開発チーム：「コンテナで動かしているので環境差異がない」→ 開発環境と本番環境が同じ構成で動く
Phase 1-4 参照
Kubernetes K8s
インフラ

大量のコンテナを自動管理するシステム。コンテナの「指揮者」で、どのサーバーでどのコンテナを動かすかを自動制御する。

開発チーム：「K8sクラスタで運用する」→ コンテナを自動管理・スケールできる基盤を使う
Phase 1-3 参照
冗長化 Redundancy
インフラ

同じ機能を持つ機器を複数用意し、1台が壊れても別の1台が引き継ぐ構成。「予備の発電機」のようなもの。

開発チーム：「DBは冗長構成にします」→ データベースを2台以上用意して障害に備える
Phase 2-2 参照
SPOF Single Point of Failure
インフラ

その1箇所が壊れるとシステム全体が止まるポイント。「橋が1本しかない川」のようなもの。構成図で1台しかないコンポーネントを探す。

開発チーム：「ここがSPOFになっている」→ このポイントが止まると全部止まる危険がある
Phase 3-4 参照
API Application Programming Interface
開発プロセス

ソフトウェア同士が会話するための「窓口」。レストランのメニュー表のように、「何を頼めば何が返ってくるか」が決まっている。

開発チーム：「APIを公開する」→ 他のシステムがデータを取得・送信できる窓口を作る
Phase 1-5 参照
REST API RESTful API
開発プロセス

Web上で最も一般的なAPIの形式。URLにアクセスしてデータをやり取りする。ブラウザでWebサイトを見る仕組みの延長線上にある。

開発チーム：「REST APIで連携する」→ HTTPベースの標準的な方法でシステム間を接続する
Phase 1-5 参照
Git Version Control
開発プロセス

ソースコードの変更履歴を管理するツール。Wordの「変更履歴」の超高機能版。誰がいつ何を変えたか全て記録される。

開発チーム：「Gitにpushした」→ コードの変更をチーム共有のリポジトリに反映した
CI/CD Continuous Integration / Delivery
開発プロセス

コードの変更を自動でテスト・ビルド・デプロイする仕組み。工場の「自動化ライン」。人手を介さず品質チェックからリリースまで行う。

開発チーム：「CI/CDパイプラインを構築する」→ 自動テスト・自動デプロイの仕組みを作る
デプロイ Deploy
開発プロセス

開発したプログラムを本番環境に配置して使えるようにすること。「新商品を棚に並べる」作業にあたる。

開発チーム：「今夜デプロイします」→ 今夜、本番環境にプログラムを配置する
マイグレーション Migration
開発プロセス

既存のシステムやデータを新しい環境に移行すること。「引っ越し」と同じで、荷物（データ）を壊さず新居（新システム）に運ぶ必要がある。

開発チーム：「データマイグレーションに2週間必要」→ 既存データの移行作業に2週間かかる
リファクタリング Refactoring
開発プロセス

動作を変えずにコードの内部構造を改善すること。家の「リフォーム」に近い。見た目（機能）は変わらないが、住みやすさ（保守性）が向上する。

開発チーム：「リファクタリングの時間がほしい」→ 今後の開発を楽にするためにコード整理をしたい
技術的負債 Technical Debt
開発プロセス

「今は楽だけど後で大変になる」コードの借金。急いで作ったツケが溜まり、将来の機能追加や修正が困難になっていく状態。

開発チーム：「技術的負債が溜まっている」→ 過去の設計判断のツケで、改修に余計な工数がかかる状態
スプリント Sprint
開発プロセス

アジャイル開発における、通常2〜4週間の開発サイクル。1スプリントごとに動くソフトウェアを作り、お客様にレビューしてもらう。

開発チーム：「次のスプリントで対応します」→ 次の2〜4週間の開発サイクルで取り組む
Phase 4-1 参照
MVP Minimum Viable Product
開発プロセス

最小限の機能で動く製品。「全機能を作ってからリリース」ではなく「最低限の機能で早くリリースし、フィードバックを得る」戦略。

開発チーム：「まずMVPで出しましょう」→ 最低限の機能で早くリリースし、反応を見てから拡張する
PoC Proof of Concept
開発プロセス

「本当にできるか」を確認するための小規模な検証。本番開発の前に技術的な実現可能性を確かめる「お試し」。

開発チーム：「まずPoCで検証したい」→ 本格開発の前に、技術的に実現可能か小さく試したい
コードレビュー Code Review
開発プロセス

他の開発者がソースコードをチェックする品質管理プロセス。論文の「査読」にあたる。バグや設計ミスを早期に発見する。

開発チーム：「レビュー待ちです」→ 他のメンバーのコードチェックを待っている状態
テスト自動化 Test Automation
開発プロセス

テストをプログラムで自動実行する仕組み。手動テストに比べ、繰り返し実行が容易で品質が安定する。初期コストは高いが長期的にはコスト減。

開発チーム：「テストを自動化したい」→ テストをプログラムで自動実行し、品質を安定させたい
認証/認可 Authentication / Authorization
セキュリティ

認証は「あなたは誰？」を確認すること（ログイン）。認可は「あなたは何ができる？」を決めること（権限管理）。セットで使われることが多い。

開発チーム：「認証はOAuthで認可はRBACで」→ ログインは外部サービス連携、権限は役割ベースで管理
Phase 2-4 参照
SSO Single Sign-On
セキュリティ

1回のログインで複数のシステムにアクセスできる仕組み。「社員証1枚で全部屋に入れる」ようなもの。

開発チーム：「SSO対応が必要」→ 既存の社内認証基盤と連携したログイン機能を作る
Phase 2-4 参照
暗号化 Encryption / SSL/TLS
セキュリティ

データを読めない形に変換して保護すること。通信の暗号化（HTTPS）と保存データの暗号化がある。手紙を「暗号文」にして送るイメージ。

開発チーム：「通信はSSL/TLSで暗号化」→ インターネット上のデータを盗み見できないように保護する
Phase 2-4 参照
脆弱性 Vulnerability
セキュリティ

セキュリティ上の弱点・穴のこと。建物でいう「鍵のかかっていない窓」。放置すると不正アクセスの入口になる。

開発チーム：「脆弱性スキャンで問題が見つかった」→ セキュリティチェックで弱点が発見された、対応が必要
Phase 2-4 参照
ゼロトラスト Zero Trust
セキュリティ

「社内ネットワークだから安全」と信用せず、すべてのアクセスを毎回検証するセキュリティモデル。「社員証を持っていても毎回本人確認する」方針。

開発チーム：「ゼロトラストモデルで設計する」→ 内部/外部に関わらず全てのアクセスを検証する方式にする
Phase 2-4 参照
WAF Web Application Firewall
セキュリティ

Webアプリへの攻撃（SQLインジェクション等）を検知・遮断する専門の防御壁。通常のファイアウォールよりWebアプリに特化している。

開発チーム：「WAFを前段に入れましょう」→ Webアプリへの攻撃を自動的にブロックする防御層を追加する
多要素認証 MFA / Multi-Factor Authentication
セキュリティ

パスワードに加えてSMSコードや指紋など複数の方法で本人確認すること。「鍵」と「暗証番号」の両方が必要な金庫のようなもの。

開発チーム：「MFAを必須にしたい」→ パスワードだけでなく追加の認証手段を求める仕様にする
データベース Database / RDB
データ

データを整理して保存・検索するための仕組み。Excelの超大規模版。RDB（リレーショナルDB）は表形式でデータを管理する。

開発チーム：「RDBかNoSQLか検討中」→ データの管理方式を比較検討している
Phase 1-2 参照
NoSQL Not Only SQL
データ

表形式（SQL）にこだわらないデータベース。大量データの高速処理や柔軟なデータ構造が得意。SNSの投稿データやIoTセンサーデータなどに適する。

開発チーム：「ログデータはNoSQLに入れる」→ 大量の非構造化データを高速に保存・検索する
キャッシュ Cache / Redis
データ

よく使うデータを高速な場所に一時保存する仕組み。「よく使う書類をデスクの上に置いておく」ようなもの。DBへのアクセスを減らし高速化する。

開発チーム：「Redisでキャッシュする」→ よく読まれるデータをメモリに置いて表示を高速化する
Phase 2-5 参照
ETL Extract, Transform, Load
データ

データを「抽出（E）→変換（T）→格納（L）」する処理。様々なシステムからデータを集めて、分析しやすい形に整えて保存する。

開発チーム：「ETLパイプラインを構築する」→ 複数システムのデータを収集・加工・格納する仕組みを作る
Phase 3-4 参照
データレイク Data Lake
データ

あらゆる種類のデータを「そのまま」大量に保存する場所。湖（レイク）のように何でも流し込める。後から必要に応じて分析する。

開発チーム：「まずデータレイクに溜めましょう」→ まず全データを一箇所に集め、後で分析方法を決める
DWH Data Warehouse
データ

分析用に整理されたデータの倉庫。データレイクの「生データ」に対し、DWHは「分析しやすく整理されたデータ」を保持する。

開発チーム：「BigQueryをDWHに使う」→ Google の分析用データベースで大規模データ分析する
バックアップ/リストア Backup / Restore
データ

データの複製を保存（バックアップ）し、障害時に復元（リストア）すること。「大事な書類のコピーを金庫に保管する」のと同じ。

開発チーム：「日次バックアップでRPO 24時間」→ 毎日データを保存するので、最悪24時間分のデータが失われる可能性
Phase 2-2 参照
メッセージキュー Message Queue / Kafka
データ

システム間でメッセージ（データ）を受け渡すための「郵便受け」。送り側と受け側が同時に動いている必要がなく、非同期で処理できる。

開発チーム：「Kafkaで非同期処理する」→ メッセージキューを使い、重い処理を後から実行する
Phase 1-5 参照
トランザクション Transaction / ACID
データ

「全部成功するか、全部失敗するか」のどちらかにする処理の単位。銀行送金で「引き落としだけ成功して入金は失敗」を防ぐ仕組み。

開発チーム：「この処理はトランザクションで管理する」→ 途中で失敗しても中途半端な状態にならないようにする
AWS Amazon Web Services
クラウド

Amazonが提供する世界最大のクラウドサービス。サーバー、DB、AI、ストレージなど200以上のサービスがある。シェアNo.1。

開発チーム：「AWSのEC2とRDSを使う」→ Amazonクラウドの仮想サーバーとデータベースサービスを使う
Phase 1-4 参照
Azure Microsoft Azure
クラウド

Microsoftが提供するクラウドサービス。Office 365やActive Directoryとの連携が強み。既にMicrosoft製品を使っている企業に人気。

開発チーム：「Azure ADと連携する」→ MicrosoftのID管理サービスでログイン認証する
Phase 1-4 参照
GCP Google Cloud Platform
クラウド

Googleが提供するクラウドサービス。データ分析（BigQuery）やAI/機械学習に強み。Googleのインフラ技術が基盤。

開発チーム：「BigQueryでデータ分析する」→ Googleの分析基盤で大規模データを高速処理する
Phase 1-4 参照
SaaS Software as a Service
クラウド

ソフトウェアをインターネット経由で利用するサービス。Gmail、Slack、Salesforceなどが該当。インストール不要でブラウザから使える。

開発チーム：「そこはSaaSで済ませましょう」→ 自分で作らず既存のクラウドサービスを利用する
Phase 1-4 参照
PaaS Platform as a Service
クラウド

アプリケーションの実行環境をクラウドで提供するサービス。サーバーの管理はお任せで、開発者はコードを書くことに集中できる。

開発チーム：「PaaS上にデプロイする」→ サーバー管理不要のクラウド環境にアプリを配置する
Phase 1-4 参照
IaaS Infrastructure as a Service
クラウド

仮想サーバーやネットワークなどのインフラをクラウドで提供。自由度は最大だが運用も自分でやる。「空き地を借りて自分で建てる」イメージ。

開発チーム：「IaaSだとOS管理も自分たちで」→ サーバーの中身まで自分たちで管理する必要がある
Phase 1-4 参照
サーバーレス Serverless / Lambda
クラウド

サーバーの存在を意識せずにプログラムを実行できるサービス。使った分だけ課金。「タクシー」のようにサーバーを必要な時だけ使う。

開発チーム：「Lambda で処理する」→ AWSのサーバーレス環境で、必要な時だけプログラムを実行する
Phase 1-3 参照
マルチクラウド Multi-Cloud
クラウド

複数のクラウド（AWS + Azure等）を組み合わせて使うこと。ベンダーロックインを避け、各クラウドの得意分野を活かす戦略。

開発チーム：「マルチクラウド構成にする」→ 複数のクラウドを組み合わせて、特定ベンダーへの依存を減らす
Phase 1-4 参照
オートスケール Auto Scaling
クラウド

アクセスの増減に合わせてサーバー台数を自動的に増減する仕組み。「混雑時にレジを自動で増やし、空いたら減らす」ようなもの。

開発チーム：「オートスケール設定してある」→ アクセス集中時に自動でサーバーが増える
Phase 2-3 参照
リージョン/AZ Region / Availability Zone
クラウド

リージョンはクラウドのデータセンターがある地域（東京、大阪等）。AZはリージョン内の独立したデータセンター群。複数AZで災害対策。

開発チーム：「東京リージョンの2AZ構成」→ 東京の2つの独立データセンターに分散配置して耐障害性を確保
Phase 2-2 参照
IaC Infrastructure as Code
クラウド

サーバーやネットワークの構成をコード（プログラム）で管理する手法。手順書を人が実行する代わりにコードが自動実行する。再現性が高い。

開発チーム：「Terraformで環境を作る」→ コードで自動的にクラウド環境を構築・管理する
SLA Service Level Agreement
クラウド

サービスの品質保証の約束。「稼働率99.9%を保証」「月間ダウンタイム43分以内」など、具体的な数値で合意する契約。

開発チーム：「SLA 99.9%で設計する」→ 年間で約8.7時間以内の停止に抑える構成で設計する
Phase 2-2 参照
DR Disaster Recovery
クラウド

災害時にシステムを復旧する計画・仕組み。「非常時の避難計画」のようなもの。別の場所にバックアップ環境を用意しておく。

開発チーム：「DR環境を大阪に構築」→ 東京のシステムが被災しても大阪で業務を継続できるようにする
Phase 2-2 参照
RPO Recovery Point Objective
データ

障害時に「どの時点のデータまで戻せるか」の目標。RPO 1時間なら、最悪でも1時間前のデータに復旧できる。

開発チーム：「RPO 0を要求されると構成が大きく変わる」→ データ損失ゼロの要件は同期レプリケーションが必要で高コスト
Phase 2-2 参照
RTO Recovery Time Objective
データ

障害発生から「何時間以内にシステムを復旧させるか」の目標。RTO 4時間なら、4時間以内にサービスを再開する。

開発チーム：「RTO 1時間だとホットスタンバイが必要」→ 1時間以内の復旧には、常に待機している予備システムが必要
Phase 2-2 参照
マイクロサービス Microservices
開発プロセス

機能ごとに独立した小さなサービスに分割するアーキテクチャ。「専門店が集まるモール」のように、各サービスが独立して動く。

開発チーム：「マイクロサービスに分割する」→ 大きなシステムを機能ごとの小さなサービスに分ける
Phase 1-3 参照
モノリス Monolith
開発プロセス

全機能を1つのアプリケーションに含む構成。「一軒家」のようにすべてが1つの建物に収まっている。シンプルだが大規模化すると管理が大変。

開発チーム：「現行はモノリスです」→ 全機能が1つのアプリに入っている（分割されていない）
Phase 1-3 参照
ミドルウェア Middleware

確認クイズ

Q1. 開発チームが「SPOFをなくしたい」と言ったとき、何をしようとしている？

SPOF（Single Point of Failure）は「その1箇所が止まるとシステム全体が止まるポイント」です。これをなくすとは、冗長化（複数台構成）して、1台が止まっても別の1台が引き継げるようにすることです。

Q2. 「技術的負債が溜まっている」とはどういう状態？

技術的負債は、過去に「急いで作った」「仮の対応で済ませた」などのツケが溜まり、新機能の追加やバグ修正に余計な工数がかかる状態を指します。

Q3. 「CDNに乗せる」の効果として正しいのは？

CDN（Content Delivery Network）は世界中のサーバーにコンテンツをキャッシュし、ユーザーに最も近い場所から配信します。表示速度の向上とメインサーバーの負荷軽減が主な効果です。
